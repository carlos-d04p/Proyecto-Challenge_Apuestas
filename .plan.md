## Fase 4 — Implementar `apps/payments/` cumpliendo el manifiesto

### 4.1 `apps/payments/models.py`
- `Wallet` (1‑a‑1 con `AUTH_USER_MODEL`):
  - `balance = DecimalField(max_digits=18, decimal_places=4, default=Decimal("0.0000"))` → **RB-PAY-01**
  - `CheckConstraint(balance__gte=0)` → **RB-PAY-06** (no sobregiro)
  - Métodos `credit(amount)` / `debit(amount)` privados, llamados solo desde servicios.
- `PaymentTransaction` (libro mayor inmutable, append‑only) → **RB-PAY-09**:
  - `id (UUID)`, `wallet (FK)`, `kind ∈ {DEPOSIT, WITHDRAWAL}`, `amount (Decimal 18,4)`,
    `balance_after (Decimal 18,4)`, `status ∈ {COMPLETED, FAILED}`, `idempotency_key (unique)`,
    `created_at (auto_now_add)`.
  - Override `save()` para impedir UPDATE (`if self.pk: raise`).
  - Sin `delete()` expuesto en el admin (`has_delete_permission=False`).
- Constantes (en `apps/payments/constants.py`):
  - `MIN_DEPOSIT = Decimal("50.0000")`, `MAX_DEPOSIT = Decimal("10000.0000")` → **RB-PAY-04**
  - `MIN_WITHDRAWAL = Decimal("100.0000")` → **RB-PAY-07**

### 4.2 `apps/payments/services.py`
Toda mutación bajo `transaction.atomic()` + `select_for_update()` sobre `Wallet` → **RB-PAY-02**.

- `deposit(user, amount, idempotency_key) -> PaymentTransaction`
  - Valida `MIN_DEPOSIT <= amount <= MAX_DEPOSIT` (`ValidationError` si no).
  - Si ya existe `PaymentTransaction` con esa `idempotency_key`, **retorna la existente** sin volver a acreditar → **RB-PAY-03**.
  - Acredita instantáneo, status `COMPLETED` → **RB-PAY-05**.
- `withdraw(user, amount, idempotency_key) -> PaymentTransaction`
  - Verifica `user.is_email_verified` → **RB-PAY-08** (si no, `PermissionDenied`).
  - Valida `amount >= MIN_WITHDRAWAL`.
  - Verifica `amount <= wallet.balance` con `select_for_update()` → **RB-PAY-06**.
  - Idempotencia idéntica al depósito → **RB-PAY-03**.
  - Debita, registra `PaymentTransaction` con `balance_after`.

### 4.3 `apps/payments/serializers.py`
- `DepositSerializer`, `WithdrawalSerializer` con `amount` como `DecimalField(max_digits=18, decimal_places=4)`.
- Validan rangos en `validate_amount`.

### 4.4 `apps/payments/views.py` (DRF API)
- `POST /api/payments/deposit/` → `DepositView` (IsAuthenticated). Lee header `Idempotency-Key` (RB-PAY-03; rechaza 400 si falta).
- `POST /api/payments/withdraw/` → `WithdrawView` (IsAuthenticated + verificación de KYC; 403 si no verificado → **RB-PAY-08**).
- `GET /api/payments/transactions/` → historial paginado del usuario (auditoría visible).
- Vistas HTML:
  - `PaymentsDashboardView` (`/payments/`) — saldo + formularios + historial.
  - El botón "Retirar" se renderiza `disabled` si `not request.user.is_email_verified` → **RB-PAY-08** (visual).
